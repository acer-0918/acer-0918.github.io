SQL & pandas

SQL 与 pandas 的基础映射

表

SQL的table，等同于pandas的DataFrame。

选列

SELECT user_id, amount
FROM orders;

等同于

df[['user_id', 'amount']]

注意：

在SQL中，一般来说;之前的所有内容作为一个完整的SQL语句进行解析和执行。

对于一般的空格（当然字符串里的和影响词汇间隔的是不可忽略的）、空行和缩进，SQL并不敏感，但最好保持好的清晰的代码习惯。大小写问题稍微复杂一点，不过保持好习惯肯定不会出错。

pandas中选择多列df[[列名列表]]，双中括号。如果选择单列df['amount']，返回的是一个Series，不是完整的二维表。

过滤行

SELECT *
FROM orders
WHERE status = 'paid';
df[df['status'] == 'paid']

注意，SELECT *中*是通配符。在只指定了行条件的情况下，指的是选中所有列（仅限符合要求的行）。

以及SQL的等号判断就是=，不是==。例如WHERE status = 'paid'

多条件过滤

SELECT *
FROM orders
WHERE status = 'paid' AND amount > 100;
df[(df['status'] == 'paid') & (df['amount'] > 100)]

AND对应&，如果是或那就是OR对应|。注意pandas中每个条件都要加括号。

排序

降序：

SELECT *
FROM orders
ORDER BY amount DESC;
df.sort_values('amount', ascending=False)

升序：

SELECT *
FROM orders
ORDER BY amount ASC;
df.sort_values('amount', ascending=True)

ASC就是ascending嘛。另外pandas中排序的默认行为是升序。

取前几行

SELECT *
FROM orders
LIMIT 5;
df.head(5)

特别地，df.head()就是默认前5行。

去重

SELECT DISTINCT city
FROM orders;
df['city'].drop_duplicates()

返回的是具体的值（Series）。如果要求统计不重复元素数量：

SELECT COUNT(DISTINCT city)
FROM orders;
df['city'].nunique()

派生新列

SELECT *,
       amount * 0.9 AS discount_amount
FROM orders;
df['discount_amount'] = df['amount'] * 0.9

pandas多条件要加括号，选多列加中括号。

SQL中命令的基本顺序通常是

SELECT ...
FROM ...
WHERE ...
ORDER BY ...
LIMIT ...

聚合与分组

总数

SELECT COUNT(*) AS order_cnt
FROM orders;
len(df)
# 或者
df.shape[0]

注意：SELECT COUNT(*) AS order_cnt当中的AS，用于对查询结果进行重命名。查询结果中，原本可能显示为COUNT(*)的列现在会显示为order_cnt。

总和

SELECT SUM(amount) AS total_amount
FROM orders;
df['amount'].sum()

均值

SELECT AVG(amount) AS avg_amount
FROM orders;
df['amount'].mean()

分组统计

SELECT city, COUNT(*) AS order_cnt
FROM orders
GROUP BY city;
df.groupby('city').size()

需要注意SQL这里，尽管没有操作city列，在后面也写出了GROUP BY city，但SELECT中依然要写进city。

更常见的是一起做：

SELECT city,
       COUNT(*) AS order_cnt,
       SUM(amount) AS total_amount,
       AVG(amount) AS avg_amount
FROM orders
GROUP BY city;
df.groupby('city')['amount'].agg(['count', 'sum', 'mean'])

特别地，pandas中这样操作常常会让city成为索引。经常恢复为普通列：

df.groupby('city')['amount'].agg(['count', 'sum', 'mean']).reset_index()

依据多列分组

SELECT city, status, COUNT(*) AS order_cnt
FROM orders
GROUP BY city, status;
df.groupby(['city', 'status']).size().reset_index(name='order_cnt')

注意：

  1. 如果不加.reset_index()，会得到多重索引MultiIndex，我们一般不太喜欢
  2. 如果不指定name参数，列名默认是0

更进一步地，上面多列分组的例子中，

df.groupby('city')['amount'].agg(['count', 'sum', 'mean']).reset_index()

.reset_index()并没有指定name参数。这是因为，多组聚合的DataFrame.reset_index()压根就没有name参数。但它们是有列名的，是agg函数指定的三个函数(‘count’, ‘sum’, ‘mean’)。如果我们需要自定义列名，也是可以的，只需要在agg中指定元组列表：

df.groupby('city')['amount']
    .agg([('order_count', 'count'), 
          ('total_amount', 'sum'), 
          ('avg_amount', 'mean')])
    .reset_index()

过滤分组结果

SELECT city, COUNT(*) AS order_cnt
FROM orders
GROUP BY city
HAVING COUNT(*) > 1;

在SQL中的一个核心区别是，

  • WHERE：分组前过滤原始行
  • HAVING：分组后过滤聚合结果

同时需要注意，WHERE和HAVING中提到的列，不一定非得需要在SELECT中提到。这点和GROUP BY不一样。

另外，虽然我们在这里已经给COUNT(*)定义了别名order_cnt，但在HAVING中最好还是不用别名，而是用原始的HAVING COUNT(*) > 1写法。SQL的语法就是这样，可以认为是各家发行版特性不同，只能学最大公约数了。

pandas则是写成两步，先聚合再过滤：

tmp = df.groupby('city').size().reset_index(name='order_cnt')
tmp[tmp['order_cnt'] > 1]

pandas groupby后面不要忘记选列。应该是df.groupby('city')['amount'].sum()

pandas中count() 和 size() 不完全一样。简单来说count()数非空值，size()数行数。

和SQL对应的话，几个计数函数的关系是这样的：

pandas 方法 SQL 语法 解释
size() COUNT(*) 统计分组中的总行数，包含空值
count() COUNT(column_name) 统计指定列的非空值数量
nunique() COUNT(DISTINCT column_name) 统计指定列的不重复非空值数量

在SQL中，分清WHERE和HAVING。

  • WHERE amount > 100：先筛订单
  • HAVING COUNT(*) > 10：先分组，再筛组

表连接

键

主键 (Primary Key)

  • 定义：表中唯一标识每一行记录的列（或列组合），确保唯一性和非空。
  • 特性：
    • 唯一标识：每行有且仅有一个唯一主键值
    • 非空约束：主键列不允许 NULL 值
    • 表级唯一：一张表只能有一个主键
  • 作用：数据完整性约束，作为记录的唯一标识。

连接键 (Join Key)

  • 定义：用于关联多张表的列，通常是外键引用另一表的主键。
  • 特性：
    • 关系桥梁：建立表间关联关系
    • 可重复：连接键值允许多行重复
    • 可多列：连接可基于多列组合
  • 作用：实现多表关联查询，维护数据关系。

join的意义是把分散在不同表里的信息，按共同字段拼起来。

内连接

内连接只保留两边都能匹配上的记录。

SELECT o.order_id, o.user_id, o.amount, u.city
FROM orders o
INNER JOIN users u
ON o.user_id = u.user_id;
df_orders.merge(df_users, on='user_id', how='inner')

SQL中直接写JOIN等同于INNER JOIN。这和pandas里面merge不定义how的默认行为一致。

SQL中如果用到了多表，最好养成写别名的好习惯FROM orders o，这样可以避免重复字段。

另外，pandas的merge也可以通过参数来解决左右表连接字段列名不同的问题。

df_orders.merge(df_users, left_on='user_id', right_on='uid', how='inner')

左连接

左连接保留左表的所有记录。右表能匹配上就补信息，匹配不上就填空值。

SELECT o.order_id, o.user_id, o.amount, u.city
FROM orders o
LEFT JOIN users u
ON o.user_id = u.user_id;
df_orders.merge(df_users, on='user_id', how='left')

一个LEFT JOIN和WHERE联动时候的经典陷阱。

SELECT *
FROM orders o
LEFT JOIN users u
ON o.user_id = u.user_id
WHERE u.city = 'Beijing';

它的效果并不是LEFT JOIN。实际上这个写法通常会把匹配不到的行也过滤掉，效果很像INNER JOIN。

这是因为WHERE u.city = 'Beijing'会把u.city IS NULL的行去掉。

如果真的想保留左表全部，只是在右表匹配时限制条件，常见写法是把条件放进ON里：

SELECT *
FROM orders o
LEFT JOIN users u
ON o.user_id = u.user_id
AND u.city = 'Beijing';
  • 放在WHERE，是在join完之后筛结果
  • 放在ON，是在匹配右表时加条件

业务、粒度和话术

一段关于join的业务话术： 我会先以订单表为主表，用 LEFT JOIN 关联用户表，通过 user_id 补充用户维度信息，比如城市。之所以用 LEFT JOIN，是为了保留所有订单，避免因为用户维表缺失而丢单。随后再按城市聚合计算订单量或 GMV。同时我会注意连接键是否重复，避免 join 后出现数据膨胀和重复计数问题。
